## [2.0.0] — 2026-05-14

### BREAKING

- **`vatPercent` and `percentAmount` now take percent values, not decimal fractions.**
  Pass `15` for 15% VAT (matches every public doc and example). Previously the math interpreted `15` as 1500% — internal-only fixtures used `0.15` while every doc, type JSDoc, and example showed `15`. The math is now aligned with the documented contract: a `15` input produces a `15.00` `cbc:Percent` in XML and a 15% line tax.
  Migration: multiply existing `vatPercent` / `percentAmount` values by 100 (e.g. `0.15` → `15`). Affects `ZATCAInvoiceLineItem` and `ZATCAInvoiceLineItemTax`.
  Affected internal math: `shared-tax-arithmetic.ts`, `shared-debit-note-arithmetic.ts`, `phase1-invoice.ts`, `phase1-credit-note.ts`. Captured golden vectors (`fixtures/_keys`) stay byte-identical — only the input mapping conversion was relocated.

### Changed

- Brand spelling normalised across docs, package metadata, source, examples, and generated TypeDoc HTML: `dokhna-tach` → `dokhna-tech` (also fixed the stray `dokhna.tach` in `CODE_OF_CONDUCT.md`).

### Fixed

- BSL 1.1 full upstream legal text body now appended to `LICENSE` verbatim from <https://mariadb.com/bsl11/>; the parameter block (Licensor, Licensed Work, Additional Use Grant, Change Date, Change License) is unchanged. Closes the corresponding v1.1.0 gap recorded under `[1.0.0]`.
- Placeholder maintainer contact emails replaced with real addresses on the `@dokhna.tech` domain: `security@dokhna.tech` (`SECURITY.md`), `licensing@dokhna.tech` (`README.md`, `LICENSES/COMMERCIAL.md`), `zatca@dokhna.tech` (`CODE_OF_CONDUCT.md`). Closes the corresponding v1.1.0 gap recorded under `[1.0.0]`.

## [1.0.0] — 2026-05-13

### Overview

First stable release of the `@dokhna-tech/zatca` family. This is the initial public, citable, semver-stable surface for Saudi Arabia ZATCA Phase 2 e-invoicing in Node.js. Four packages ship in lockstep at `1.0.0`:

| Package | Purpose |
|---------|---------|
| `@dokhna-tech/zatca` | Core: XML build, XMLDSig signing, ZATCA SHA-256 hashing, TLV QR, API client, onboarding |
| `@dokhna-tech/zatca-storage-memory` | In-memory `StorageAdapter` for tests/dev |
| `@dokhna-tech/zatca-storage-mongo` | MongoDB (Mongoose) `StorageAdapter` |
| `@dokhna-tech/zatca-storage-postgres` | PostgreSQL (raw `pg`) `StorageAdapter` |

### Public API

**Invoice builders (6):**
- `buildStandardTaxInvoice`
- `buildStandardCreditNote`
- `buildStandardDebitNote`
- `buildSimplifiedTaxInvoice`
- `buildSimplifiedCreditNote`
- `buildPhase1Invoice`
- `buildPhase1CreditNote`

**Crypto / XML / QR:**
- `signInvoiceXml`, `signedPropertiesHash`, `invoiceHash` (ZATCA SHA-256, byte-identical to ZATCA reference)
- `buildQR` (TLV, base64-encoded)
- `generateKeyPair`, `generateCsr`, `probeOpenSSL`
- 3 golden vectors with byte-identity assertions: standard tax invoice, simplified tax invoice, credit note

**API client (ZATCA endpoints):**
- `issueComplianceCertificate`
- `runComplianceTests`
- `issueProductionCsids`
- `clearInvoice`, `reportInvoice`
- `checkInvoiceStatus`, `cancelInvoice`
- Native `fetch` with configurable retry, exponential backoff, and timeout
- Sandbox / simulation / production environment URLs baked in

**Onboarding:**
- `onboard()` — full end-to-end orchestration: generate keys → CSR → compliance CSID → run compliance tests → production CSIDs

**Storage:**
- `StorageAdapter` contract with conformance suite shared across all three adapters
- 13 conformance tests pass on memory, mongo, and postgres

**Validation:**
- Zod schemas for every public payload
- CRN, VAT, UUID, and address validators

### Quality bar

- 3 moderate `pnpm audit` advisories in dev-only paths (vitest → esbuild, vitest → vite) — not in the runtime tree. The third (fast-xml-parser ≤4.5.0) is in the runtime path; tracked for v1.0.1 bump.
